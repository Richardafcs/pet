# 开发路线图

## Hackathon 开发策略

先做可演示闭环，再替换真实服务。不要一开始把时间耗在不稳定 OCR、账户系统或复杂后端上。

## 推荐分工

| 角色 | 负责内容 |
| --- | --- |
| Product/Design | 用户流程、文案、演示脚本、宠物状态设计 |
| Frontend | 页面、组件、状态管理、响应式 |
| Data/Algorithm | 数据模型、计划算法、宠物状态计算 |
| AI/OCR | mock 识别、OCR adapter、样本数据 |
| QA/Demo | 测试路径、演示数据、部署 |

小团队可合并角色。

## 0-2 小时：项目初始化

目标：

- 初始化前端项目。
- 建立基础页面路由。
- 加入 demo seed 数据。
- 建立数据类型。

交付：

- 可打开首页。
- 可以看到宠物占位和库存占位。
- 本地数据结构通过类型检查。

## 2-6 小时：核心数据闭环

目标：

- 手动添加物品。
- 库存列表。
- 计划算法。
- 行动记录。
- 宠物状态更新。

交付：

- 用户能手动添加菠菜、酸奶、米饭。
- 今日计划能推荐菠菜。
- 标记完成后宠物状态变化。

## 6-10 小时：小票和照片演示

目标：

- 上传小票 UI。
- mock 小票识别结果。
- 上传照片 UI。
- mock 图片识别结果。
- 候选项确认页面。

交付：

- 使用预置小票图片生成候选项。
- 用户确认后进入库存。
- 低置信字段显示确认提示。

## 10-14 小时：体验完善

目标：

- Dashboard 完成。
- 风险标签和筛选。
- 宠物视觉状态。
- Impact 页面。
- 空状态、错误状态。

交付：

- 评委能在 3 分钟内看懂问题和方案。
- 首页能显示今日重点和宠物反馈。
- 成果指标可展示减少浪费价值。

## 14-18 小时：测试和稳定

目标：

- 单元测试覆盖计划算法和宠物状态。
- E2E 跑通核心路径。
- 修复移动端布局。
- 准备演示脚本。

交付：

- 本地完整工作区中 `npm test` 或等价命令通过；公开最小仓库至少需要 `typecheck`、`lint`、`build` 通过。
- 演示路径可重复执行。
- 部署链接可访问。

## 18-24 小时：增强和 polish

可选增强：

- 接入真实 OCR 试验。
- 条码查询 Open Food Facts。
- 菜谱/搭配建议。
- 宠物动画。
- 浏览器通知。
- 分享家庭库存。

注意：增强功能不能破坏核心 mock 演示路径。

## 里程碑

### Milestone 1: Documentation Ready

- 产品需求完成。
- 技术设计完成。
- 数据模型完成。
- 测试计划完成。

### Milestone 2: MVP Loop

- 手动添加。
- 计划生成。
- 行动完成。
- 宠物反馈。

### Milestone 3: Recognition Demo

- 小票 mock 识别。
- 照片 mock 识别。
- 用户确认流程。

### Milestone 4: Demo Ready

- 部署完成。
- 测试通过。
- 演示脚本完成。
- 参考来源可解释。

## 演示脚本

1. 展示问题：冰箱里有临期食物，用户经常忘记。
2. 上传小票：系统识别菠菜、酸奶、米饭。
3. 用户确认：低置信字段可编辑。
4. Dashboard：宠物提醒今天先用菠菜。
5. 完成行动：标记菠菜已使用。
6. 宠物恢复：健康和心情上升。
7. Impact：展示避免浪费的物品数量。
8. 结尾：说明未来可接入真实 OCR、条码和共享库存。
