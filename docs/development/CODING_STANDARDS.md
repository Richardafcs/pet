# 编码规范

## TypeScript

- 开启 `strict`。
- 不使用 `any`，除非 adapter 边界必须接收第三方返回值；使用后立即用 Zod 或手写 parser 收窄。
- 领域类型集中在 `src/types/domain.ts`。
- 时间使用 ISO 日期字符串，日期计算通过工具函数封装。
- 置信度使用 `0-1` 浮点数，进入 UI 前格式化成百分比。

## React

- 组件只负责展示和用户交互。
- 复杂业务逻辑放到 pure functions。
- 表单输入应有明确 label 和错误信息。
- 不在组件中硬编码长食品安全说明，统一从 copy 常量读取。
- 列表项需要稳定 `key`。

## 状态管理

- 小型 MVP 可以使用 Zustand。
- 全局状态只放跨页面需要的数据。
- 表单临时状态放组件内部。
- 计划结果可由库存派生，不要不必要地重复持久化。

## 日期处理

禁止手写脆弱日期计算。建议：

- 使用 `date-fns`。
- 内部保存 `YYYY-MM-DD` 或 ISO timestamp。
- UI 根据用户 locale 格式化。

常用工具函数：

```ts
daysUntil(date: IsoDate, today: IsoDate): number
toIsoDate(date: Date): IsoDate
parseOptionalIsoDate(value: unknown): IsoDate | undefined
```

## 样式

- UI 应偏应用工具，不做营销页。
- 风险状态必须有文字标签，不只靠颜色。
- 移动端优先，主要操作应容易单手完成。
- 不使用过度装饰背景。
- 宠物动画要轻量，不能影响页面可读性。

## 文案

推荐：

- “建议优先使用”
- “请确认识别结果”
- “如果不确定安全性，请遵循当地食品安全建议”

禁止：

- “这个一定安全”
- “过了这个日期必须丢弃”
- “你浪费了食物”

## 错误处理

每个错误都应有恢复路径：

- 上传失败 -> 重试或手动添加。
- 识别失败 -> 手动添加。
- 存储失败 -> 告知用户并允许导出/重试。
- 外部服务失败 -> 使用 mock 或手动流程。

## 代码注释

只在业务规则不显然时加注释，例如计划评分权重、宠物状态衰减规则。不要注释显而易见的赋值。
