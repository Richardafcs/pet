# 项目范围

## 重要范围修正

当前项目不再以“精确记录所有食物库存和每次消耗”为目标。这个方向在真实使用中负担过高，也容易被照片识别、多人共用冰箱和用户忘记打卡击穿。

新的 MVP 范围是：

> 用低摩擦方式帮助用户避免重复购买，并每天处理最可能被浪费的少数高风险食物。

因此项目优先级从“完整库存管理”调整为：

1. 购买前提醒：识别重复购买和过量购买风险。
2. 每日 rescue decision：只展示 1 到 3 个最重要行动。
3. 粗粒度 check-in：使用 opened、some left、almost gone、finished 等状态替代精确剩余量。
4. 宠物反馈：让避免浪费、重复购买、丢弃和忽略行为产生可见后果。

完整产品转向和前三名目标计划见 [PRODUCT_PIVOT_AND_TOP3_PLAN.md](PRODUCT_PIVOT_AND_TOP3_PLAN.md)。

## 背景

家庭食物浪费常见原因包括：买完忘记、日期标签理解混乱、不会安排临期物品、剩菜缺少使用提醒、冰箱库存不可见。`pet` 试图用“库存管理 + 使用计划 + 虚拟宠物反馈”的方式，把减少浪费变成一个每天可以完成的小循环。

## 项目目标

- 让用户更容易知道家里有什么食物。
- 按临期程度、数量和可用场景推荐使用顺序。
- 用虚拟宠物状态给用户即时反馈，提升持续使用动力。
- 在 hackathon 中演示一个从小票/照片输入到计划和宠物反馈的完整体验。
- 为后续真实开发留下清晰的数据模型、模块边界和测试标准。

## 目标用户

- 经常买菜但容易忘记库存的人。
- 合租、家庭或学生宿舍，需要共享食物库存的人。
- 想减少浪费但不想使用复杂库存软件的人。
- hackathon 评委和观众：需要在短时间内理解问题、方案和影响。

## MVP 范围

MVP 必须包含：

- 手动添加食物。
- 上传小票图片并生成可编辑的识别结果。
- 上传食物照片并生成可编辑的识别结果。
- 库存列表，支持临期排序。
- 今日计划和本周计划。
- 完成、冷冻、分享/捐赠、丢弃四类行动。
- 宠物状态变化。
- 简单成果指标：已使用物品、避免浪费估算、连续完成天数。

## 非目标

MVP 阶段不做：

- 精确食品安全判断。
- 医疗、营养或饮食治疗建议。
- 复杂多用户权限系统。
- 真实支付、购物或自动下单。
- 完整冰箱硬件集成。
- 保证 OCR 或图像识别 100% 准确。

## 设计原则

- 先确认，再自动化：识别结果必须由用户确认。
- 先预防，再处理：优先安排使用和保存，而不是只统计丢弃。
- 低负担：添加物品和完成行动都应在 30 秒内完成。
- 反馈温和：宠物可以状态变差，但不应羞辱用户。
- 可解释：计划必须说明为什么推荐某个物品。

## 风险和约束

- 小票 OCR 对不同商店格式、缩写、折扣行和税费行很敏感。
- 食物照片识别可能混淆相似物品，需要用户校正。
- 建议使用日期取决于储存条件，不能只靠类别推断。
- 日期标签含义不同，“Best if Used By” 多数是质量提示，不等同于安全截止。
- hackathon 时间有限，应先实现 mock 数据和手动确认闭环。
